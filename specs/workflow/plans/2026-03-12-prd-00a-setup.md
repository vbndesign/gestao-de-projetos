# PRD-00a — Setup: Implementation Plan

## Overview

Inicializar o projeto Next.js 16 com TypeScript strict, Tailwind CSS 4.x e shadcn/ui na raiz do repositório existente (que já contém `specs/`, `CLAUDE.md` e `.gitignore`). Ao final, `pnpm dev` roda e `pnpm build` passa sem erros, com a estrutura de pastas completa para os PRDs seguintes.

## Current State Analysis

O repositório contém apenas documentação — zero código de aplicação:

```
.gitignore
CLAUDE.md
specs/foundation/         — 7 arquivos (00 a 06)
specs/prds/               — roadmap, template, prd-00a-setup
specs/workflow/            — prompts, research, plans
```

Não existe `package.json`, `tsconfig.json`, `next.config.ts`, `src/`, nem dependências instaladas.

### Key Discoveries:
- O repo já tem `.gitignore` — o `create next-app` pode sobrescrever (`research:2026-03-12-prd-00a-setup.md`)
- A spec pina Next.js 16.1.6 — o `create next-app` instala a latest do 16.x (`specs/foundation/03_arquitetura.md`)
- Tailwind v4 usa `@import "tailwindcss"` — não `@tailwind base/components/utilities` (`prd-00a-setup.md:76`)
- shadcn/ui com Tailwind v4 requer `shadcn@latest` — versões anteriores não suportam v4 (`prd-00a-setup.md:85`)

## Desired End State

Após a execução deste plano:

1. `pnpm dev` inicia o servidor local em `localhost:3000` sem erros
2. `pnpm build` completa com zero erros de TypeScript
3. Página placeholder renderiza com estilos Tailwind aplicados
4. Estrutura `src/` contém todas as camadas: `app/`, `components/ui/`, `actions/`, `services/`, `queries/`, `lib/`, `types/schemas/`
5. `@/` resolve imports corretamente (ex: `import { cn } from "@/lib/utils"`)
6. `components.json` do shadcn/ui existe e está configurado
7. TypeScript em modo strict
8. `<html lang="pt-BR">` com fonte Inter e `antialiased`

## What We're NOT Doing

- ❌ Prisma, banco de dados, variáveis de ambiente de banco (→ PRD-00b)
- ❌ Supabase Auth, `proxy.ts`, `requireAuth()` (→ PRD-00b)
- ❌ Layouts `(internal)` e `(portal)` (→ PRD-00b)
- ❌ Deploy, `vercel.json`, configuração de env vars (→ PRD posterior)
- ❌ Componentes de domínio, formulários, schemas Zod (→ PRDs seguintes)
- ❌ ESLint customizado além do padrão `eslint-config-next`

## Implementation Approach

Abordagem sequencial em 4 fases, cada uma com ponto de verificação. A ordem respeita dependências: Next.js primeiro (gera o projeto base), shadcn segundo (depende do projeto existir), ajustes terceiro (customiza o que foi gerado), validação por último.

## Phase 1: Inicialização Next.js

### Overview
Rodar `create next-app` na raiz do repo, preservando os arquivos existentes (`specs/`, `CLAUDE.md`, `.gitignore`).

### Changes Required:

#### 1. Preservar `.gitignore` existente

Antes de rodar o `create next-app`, salvar o conteúdo do `.gitignore` atual para fazer merge depois.

```bash
cp .gitignore .gitignore.backup
```

#### 2. Inicializar o projeto

**Comando** (definido em `prd-00a-setup.md:66`):

```bash
pnpm create next-app . --typescript --tailwind --app --src-dir --import-alias "@/*" --use-pnpm
```

**Notas:**
- O flag `.` cria na raiz (não gera subpasta)
- `--typescript` habilita TypeScript
- `--tailwind` instala Tailwind CSS
- `--app` habilita App Router
- `--src-dir` coloca código em `src/`
- `--import-alias "@/*"` configura path alias
- `--use-pnpm` usa pnpm como package manager

#### 3. Merge do `.gitignore`

Após o `create next-app`:
- Comparar `.gitignore` gerado com `.gitignore.backup`
- Fazer merge manual: manter entradas existentes + adicionar novas do Next.js
- Remover `.gitignore.backup`

#### 4. Verificar versão do Next.js

Conferir no `package.json` se a versão instalada é compatível com 16.x. Se o `create next-app` instalar uma versão diferente da 16.1.6 pinada nas specs, avaliar:
- Se é 16.x.x → aceitar (patch/minor differences são OK)
- Se é 15.x ou 17.x → ajustar `package.json` e rodar `pnpm install`

#### Automated Verification:
- [x] `package.json` existe na raiz com Next.js 16.x como dependência

- [x] `pnpm install` roda sem erros (dependências instaladas)
- [x] `.gitignore` contém entradas do Next.js E entradas originais preservadas

#### Manual Verification:
- [ ] `specs/` e `CLAUDE.md` continuam intactos após o init

## Phase 2: shadcn/ui

### Overview
Instalar e configurar o shadcn/ui com as preferências definidas no PRD.

### Changes Required:

#### 1. Executar shadcn init

**Comando** (definido em `prd-00a-setup.md:79`):

```bash
pnpm dlx shadcn@latest init
```

**Configuração interativa** (ou via flags se disponíveis):
- Style: **New York**
- Base color: **Neutral** (ou Zinc)
- CSS variables: **sim**

#### 2. Verificar arquivos gerados

O `shadcn init` deve gerar:
- `components.json` — configuração do shadcn na raiz
- `src/lib/utils.ts` — `cn()` helper (merge de `clsx` + `tailwind-merge`)

Verificar que:
- `components.json` tem os paths corretos (`@/components`, `@/lib/utils`)
- `src/lib/utils.ts` exporta `cn()`
- Dependências foram adicionadas ao `package.json` (`clsx`, `tailwind-merge`, etc.)

#### 3. Verificar `globals.css`

O `shadcn init` modifica `src/app/globals.css` para adicionar variáveis CSS. Verificar que:
- A diretiva `@import "tailwindcss"` está presente (sintaxe Tailwind v4)
- **NÃO** usa `@tailwind base; @tailwind components; @tailwind utilities;` (sintaxe v3)
- Variáveis CSS do tema shadcn foram adicionadas (`:root` e `.dark`)

Se o `shadcn init` gerar sintaxe v3, corrigir manualmente para v4.

#### Automated Verification:

- [x] `pnpm build` passa sem erros após o shadcn init

#### Manual Verification:

## Phase 3: Ajustes e Estrutura

### Overview
Customizar o layout raiz conforme a spec, criar pastas vazias com `.gitkeep`, e garantir que todas as configurações estão corretas.

### Changes Required:

#### 1. Customizar `src/app/layout.tsx`

Editar o layout gerado pelo `create next-app` para seguir a spec (`prd-00a-setup.md:93-97`):

```tsx
import type { Metadata } from "next";
import { Inter } from "next/font/google";
import "./globals.css";

const inter = Inter({
  subsets: ["latin"],
});

export const metadata: Metadata = {
  title: "Gestão de Projetos",
  description: "Sistema de gestão de projetos",
};

export default function RootLayout({
  children,
}: Readonly<{
  children: React.ReactNode;
}>) {
  return (
    <html lang="pt-BR">
      <body className={`${inter.className} antialiased`}>
        {children}
      </body>
    </html>
  );
}
```

**Diferenças-chave vs. o gerado pelo `create next-app`:**
- `lang="pt-BR"` (não `"en"`)
- Fonte Inter explícita via `next/font/google`
- `antialiased` no body
- Metadata em português

#### 2. Customizar `src/app/page.tsx`

Página placeholder simples para validar o setup:

```tsx
export default function Home() {
  return (
    <main className="flex min-h-screen flex-col items-center justify-center p-24">
      <h1 className="text-4xl font-bold">Gestão de Projetos</h1>
      <p className="mt-4 text-lg text-muted-foreground">
        Setup funcionando — Next.js + Tailwind + shadcn/ui
      </p>
    </main>
  );
}
```

Usa classes Tailwind (`flex`, `text-4xl`, etc.) e a variável `text-muted-foreground` do shadcn para validar que ambos funcionam.

#### 3. Criar pastas vazias com `.gitkeep`

Conforme `prd-00a-setup.md:40-48`:

```bash
mkdir -p src/actions src/services src/queries src/types/schemas src/components/ui
touch src/actions/.gitkeep
touch src/services/.gitkeep
touch src/queries/.gitkeep
touch src/types/schemas/.gitkeep
```

**Nota:** `src/components/ui/.gitkeep` — se o `shadcn init` já criou a pasta `src/components/ui/`, o `.gitkeep` só é necessário se ela estiver vazia. Se o shadcn já colocou componentes lá, não precisa do `.gitkeep`.

#### 4. Verificar TypeScript strict

Confirmar no `tsconfig.json` que `strict: true` está habilitado. O `create next-app --typescript` geralmente já configura isso, mas verificar.

## Riscos e Mitigações

| Risco | Probabilidade | Mitigação |
|---|---|---|
| `.gitignore` sobrescrito pelo `create next-app` | Alta | Backup + merge manual (Phase 1) |
| Next.js versão diferente de 16.1.6 | Média | Aceitar 16.x.x; ajustar se for 15.x/17.x |
| shadcn gera sintaxe Tailwind v3 | Baixa | Corrigir `globals.css` manualmente (Phase 2) |
| Conflito shadcn + Tailwind v4 | Baixa | `shadcn@latest` suporta v4; fallback: config manual |
