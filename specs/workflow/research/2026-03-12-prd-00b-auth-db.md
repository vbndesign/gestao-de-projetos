#### Convenção de nomes
- Models em **PascalCase**, campos em **snake_case**
- `@@map("nome_tabela")` para garantir tabelas PostgreSQL em snake_case plural
- Todos os IDs: `String @id @default(uuid()) @db.Uuid`
- Timestamps: `created_at @default(now())` + `updated_at @updatedAt`
- Campos decimais: `@db.Decimal(8, 2)` (horas)
- Campos texto longo: `@db.Text`
- Campos data (sem hora): `@db.Date`

#### Cascatas

| Ação | Comportamento |
|---|---|
| Deletar Cliente | Cascade → Projeto → tudo abaixo |
| Deletar Projeto | Cascade → Fase, registros operacionais, timeline |
| Deletar Fase | Cascade → TarefaPlanejada, LancamentoHoras |
| Deletar Fase (em registros operacionais) | SetNull → `fase_id` em Reuniao, Decisao, Checkpoint, Pendencia, Documento, MudancaDirecao, EventoTimeline |
| Deletar Reunião (em Decisao) | SetNull → `reuniao_id` |

### 3. Infraestrutura lib/ — spec-only

#### `src/lib/db.ts`
- `import 'server-only'` — bloqueia import em Client Components
- Singleton via `globalThis` para evitar múltiplas instâncias no hot reload de dev
- Prisma 5.22.x — sem driver adapters
- Log de queries apenas em `NODE_ENV === 'development'`
- Exporta `db` como `const`

Padrão definido em `specs/foundation/04_nextjs.md` (seção `server-only`):
```ts
const globalForPrisma = globalThis as unknown as { prisma: PrismaClient }
export const db = globalForPrisma.prisma ?? new PrismaClient({ log: ... })
if (process.env.NODE_ENV !== 'production') globalForPrisma.prisma = db
```

#### `src/lib/supabase.ts`
- Duas factories exportadas:
  - **`createServerSupabaseClient()`** — usa `createServerClient` de `@supabase/ssr`; implementa `getAll` + `setAll` nos cookies; `setAll` precisa de `try/catch` (Server Components não podem setar cookies)
  - **`createBrowserSupabaseClient()`** — usa `createBrowserClient` de `@supabase/ssr`; sem acesso a cookies
- Ambas usam `NEXT_PUBLIC_SUPABASE_URL` e `NEXT_PUBLIC_SUPABASE_ANON_KEY`

#### `src/lib/auth.ts`
- `import 'server-only'`
- Exportado como **`const`** (não `function`) — requisito para `React.cache()`
- Envolto em `cache()` do React — deduplicado no mesmo render pass
- Cria Supabase server client internamente (acesso direto a `cookies()`, independente de `lib/supabase.ts`)
- Chama `supabase.auth.getUser()` — validação real (não otimista)
- Sem sessão: `redirect('/login')`
- Com sessão: retorna `user` (tipo `User` de `@supabase/supabase-js`)

Padrão definido em `specs/foundation/04_nextjs.md`:
```ts
export const requireAuth = cache(async () => {
  const cookieStore = await cookies()
  const supabase = createServerClient(URL, ANON_KEY, {
    cookies: { getAll: () => cookieStore.getAll() }
  })
  const { data: { user } } = await supabase.auth.getUser()
  if (!user) redirect('/login')
  return user
})
```

### 4. Proteção de rotas — spec-only

#### `src/proxy.ts`
- Arquivo `proxy.ts` na raiz de `src/` — **não** `middleware.ts` (renomeado no Next.js 16.x)
- Exporta função `proxy()` — **não** `middleware()`
- Verificação **otimista**: apenas verifica existência do cookie `sb-access-token`
- Rotas internas protegidas: `/dashboard`, `/projetos`, `/clientes`
- Cookie ausente em rota interna → `redirect('/login')`
- `config.matcher` exclui: `_next/static`, `_next/image`, `favicon.ico`, `sitemap.xml`, `robots.txt`
- Validação real da sessão: responsabilidade do layout `(internal)` via `requireAuth()` e das Server Actions

Duas camadas de proteção (definido em `specs/foundation/04_nextjs.md`):
```
proxy.ts          → verifica existência do cookie (otimista, rápido)
(internal)/layout → confirma validade real da sessão via Supabase
actions           → requireAuth() em toda mutação (layout não re-renderiza)
```

### 5. Auth OAuth flow — spec-only

#### `src/app/auth/callback/route.ts`
- Route Handler GET
- Extrai `code` e `next` (default: `/dashboard`) dos `searchParams`
- Cria Supabase server client com `getAll` + `setAll` nos cookies
- Troca `code` por sessão: `supabase.auth.exchangeCodeForSession(code)`
- Sucesso: `redirect(next)`
- Falha ou sem `code`: `redirect('/login?error=auth_callback_failed')`

Fluxo OAuth completo:
```
1. Usuário clica "Entrar com Google" → /login
2. signInWithOAuth({ provider: 'google', redirectTo: origin + '/auth/callback' })
3. Supabase → Google OAuth
4. Google → /auth/callback?code=...
5. Route Handler troca code por sessão (seta cookie)
6. redirect('/dashboard')
```

#### `src/app/login/page.tsx`
- Server Component (sem `"use client"`)
- UI mínima: título + botão "Entrar com Google" (Client Component inline ou separado para interatividade)
- Botão chama `supabase.auth.signInWithOAuth({ provider: 'google', options: { redirectTo } })`
- Exibe mensagem de erro se searchParam `?error` presente
- Sem formulário de email/senha no MVP

## Design & Decisions

### Duas camadas de proteção de rota
O design separa intencionalmente a proteção em dois níveis:
1. `proxy.ts` — verificação otimista (cookie existe?) antes do render, sem DB
2. `(internal)/layout.tsx` + `requireAuth()` — validação real com Supabase no render

O layout **não re-renderiza** em navegações subsequentes no mesmo grupo de rotas.
Por isso, Server Actions sempre chamam `requireAuth()` independentemente do layout.

### `requireAuth()` como `const` com `React.cache()`
Exportado como `const` (não `function`) é requisito para que `React.cache()` funcione.
`React.cache()` deduplica chamadas no mesmo render pass — layout + page chamando `requireAuth()`
resulta em apenas 1 chamada ao Supabase.

### Prisma 5.22.x sem driver adapters
Prisma 6.x e 7.x exigem driver adapters obrigatoriamente — incompatível com o perfil do projeto.
Versão 5.22.x mantém a API familiar com conexão direta PostgreSQL via URL.

### `lib/supabase.ts` vs `lib/auth.ts`
`requireAuth()` em `lib/auth.ts` cria seu próprio Supabase client internamente
(não reutiliza `lib/supabase.ts`) para ter acesso direto a `cookies()` de `next/headers`
com controle total sobre a deduplicação via `React.cache()`.

### RLS desabilitado
Row Level Security do Supabase não é usado — regras de domínio ficam em `services/`,
a autenticação é verificada nas Server Actions. RLS seria para isolamento multi-tenant.

### Não usa `middleware.ts`
Next.js 16.x renomeou `middleware.ts` para `proxy.ts` e `middleware()` para `proxy()`.
Ver `specs/foundation/04_nextjs.md` para o padrão completo.
