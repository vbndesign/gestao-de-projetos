# Research: PRD-03 — Tarefas CRUD

**Date**: 2026-03-13T14:00:00-03:00
**Git Commit**: 4574a2e
**Branch**: feature/tarefas-prd-03
**Repository**: gestao-projetos
**Research Mode**: Full-stack (schema parcial existe, backend e UI a criar)

## Research Question

Levantar todo o contexto necessário para planejar a implementação do PRD-03 — CRUD completo de Tarefas Planejadas, integrado à aba Fases existente.

## Summary

PRD-03 é um PRD **full-stack** que cobre schema Zod, actions, update de query e UI integrada ao `fases-manager.tsx` existente. O modelo `TarefaPlanejada` já existe no Prisma schema com migration aplicada. A aba Fases (`fases-manager.tsx`) está funcional — PRD-03 a estende com tarefas aninhadas dentro de cada fase. Uma decisão técnica importante: `tempo_estimado_horas` é `Decimal` no Prisma, tipo não serializável nativamente pelo RSC — requer conversão explícita em `page.tsx`.

## Detailed Findings

### 1. O que já existe (schema Prisma)

**Modelo `TarefaPlanejada`** (`prisma/schema.prisma`):
```prisma
model TarefaPlanejada {
  id                    String       @id @default(uuid()) @db.Uuid
  fase_id               String       @db.Uuid
  titulo                String
  descricao             String?      @db.Text
  tempo_estimado_horas  Decimal?     @db.Decimal(8, 2)
  ordem                 Int
  status                StatusTarefa
  created_at            DateTime     @default(now())
  updated_at            DateTime     @updatedAt

  fase                  Fase         @relation(fields: [fase_id], references: [id], onDelete: Cascade)

  @@index([fase_id])
  @@map("tarefas_planejadas")
}
```

**Enum `StatusTarefa`**: `nao_iniciada` · `em_andamento` · `concluida` · `cancelada`

Migration já aplicada no Supabase. Nenhum schema change necessário.

### 2. Estado atual de `fases-manager.tsx`

O `fases-manager.tsx` existente:
- Renderiza fases em `DndContext` + `SortableContext` (reordenação de fases)
- Cada fase é um `SortableFaseCard` com drag handle (exceto Fase Geral)
- CRUD de fases via `FaseFormDialog` + `AlertDialog`
- `FaseData` type local não inclui tarefas (PRD-02c não implementou tarefas)
- `FasesManager` recebe `projetoId` e `fases` como props


### 4. Estado atual de `fases/page.tsx`

Precisa ser atualizado para converter `tempo_estimado_horas: Decimal` → `number` (ver seção 5).

### 5. Decisão técnica: Decimal → number para RSC

`tempo_estimado_horas` é `Prisma.Decimal` no retorno da query. Ao passar via RSC boundary para um Client Component, React serializa as props via JSON. `Prisma.Decimal` tem `toJSON()` que retorna uma string (ex: `"2.50"`), então o Client Component recebe uma **string**, não um número.

Para manter a tipagem limpa como `number | null` no Client Component, é necessário converter explicitamente em `page.tsx`:

```ts
const fases = fasesRaw.map(fase => ({
  ...fase,
  tarefas: fase.tarefas.map(t => ({
    ...t,
    tempo_estimado_horas: t.tempo_estimado_horas !== null
      ? Number(t.tempo_estimado_horas)
      : null,
  })),
}))
```

Isso é uma modificação menor em `page.tsx` mas necessária para type safety.

### 6. Padrões de actions existentes

**Padrão `criarFaseAction`** — referência para `criarTarefaAction`:
```ts
const maxOrdem = await db.fase.aggregate({
  where: { projeto_id: projetoId },
  _max: { ordem: true },
})
const novaOrdem = (maxOrdem._max.ordem ?? 0) + 1
await db.fase.create({ data: { ...parsed.data, projeto_id: projetoId, ordem: novaOrdem, ... } })
revalidatePath('/projetos/' + projetoId)
```

**Padrão `editarFaseAction`** — recupera `projeto_id` via `select` no `update`:
```ts
const fase = await db.fase.update({
  where: { id: faseId }, data: parsed.data,
  select: { projeto_id: true },
})
revalidatePath('/projetos/' + fase.projeto_id)
```

**Padrão `reordenarFasesAction`** — usa `$transaction` array-style (compatível com pgbouncer):
```ts
await db.$transaction(
  fasesOrdenadas.map(f => db.fase.update({ where: { id: f.id }, data: { ordem: f.ordem } }))
)
```

Para tarefas: mesmos padrões, substituindo `fase_id` e `TarefaPlanejada`.

Para `excluirTarefaAction`: precisa de `projeto_id` para `revalidatePath`. Usar `findUnique` antes de `delete`:
```ts
const tarefa = await db.tarefaPlanejada.findUnique({
  where: { id: tarefaId },
  select: { fase: { select: { projeto_id: true } } },
})
await db.tarefaPlanejada.delete({ where: { id: tarefaId } })
revalidatePath(`/projetos/${tarefa.fase.projeto_id}/fases`)
```

### 7. Padrão de DnD aninhado com @dnd-kit

O `fases-manager.tsx` já usa `DndContext` no nível de `FasesManager` para fases. Para tarefas dentro de cada fase, cada `SortableFaseCard` terá seu próprio `DndContext` aninhado.

`@dnd-kit` suporta `DndContext` aninhados — cada contexto captura eventos independentemente, sem propagação para o pai. Isso evita conflitos entre o DnD de fases e o DnD de tarefas.

**Regra anti-hydration mismatch** (aprendida no PRD-02c): passar `id={useId()}` ao `DndContext` para ID estável entre servidor e cliente.

## Regras de negócio

| Regra | Implementação |
|-------|--------------|
| Tarefas reordenáveis dentro da fase | DnD aninhado por fase + `reordenarTarefasAction` |
| Qualquer tarefa pode ser excluída | Sem restrições (sem equivalente da Fase Geral) |
| Tarefas em cascata com a fase | `onDelete: Cascade` no schema (já existe) |
| Tarefas não geram eventos de timeline | Actions não chamam nenhum service de timeline |
| `tempo_estimado_horas` é opcional | Schema: `.nullable().optional()`, form: `setValueAs` |

## Design & Decisions

- **Decimal conversion**: converter em `page.tsx` antes de passar ao Client Component — evita ambiguidade de tipo string vs number no componente
- **Sem `tarefa.queries.ts`**: arquivo standalone criado apenas quando portal precisar (PRD-08b) — pragmático para MVP
- **DnD aninhado por `SortableFaseCard`**: cada fase tem seu `DndContext` isolado — sem conflito com DnD de fases
- **`useTransition` separado no `SortableFaseCard`**: transições de tarefa isoladas das transições de fase do componente pai
- **`useId()` para DnD**: padrão aprendido no PRD-02c, aplicado ao inner `DndContext` de cada fase
- **`zodResolver as any`**: padrão estabelecido no PRD-01, aplicado ao formulário de tarefa
- **`setValueAs` para `tempo_estimado_horas`**: mesmo padrão das datas no PRD-02b, adaptado para number
